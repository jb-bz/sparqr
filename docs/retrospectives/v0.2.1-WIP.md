# v0.2.1 — Status check (work-in-progress)

**This is NOT a finalized retrospective.** It's a running status check because v0.2.1 has been attempted multiple times. This document captures all attempts honestly so the next attempt (or v0.2.2) doesn't repeat the mistakes.

## Update 2026-06-20 (this session)

### What we actually accomplished this session

1. **Discovered a major v0.2.0 bug** — `lib/*.sh` files had sentinel-var guards against double-sourcing. Bash functions don't carry across `exec` boundaries, but env vars do. So when `bin/sparc` sourced `lib/kanban.sh` (setting `SPARC_KANBAN_LOADED=1`) and then `exec`'d `bin/sparc-init`, the child process saw the env var but had no function definitions. Result: every dispatched subcommand was failing with "command not found" errors. This was masked in the unit tests because they don't go through the `bin/sparc` dispatcher.

2. **Fixed the lib/ double-source guards** — removed the guard from 7 files: `lib/kanban.sh`, `lib/stages.sh`, `lib/validators.sh`, `lib/artifacts.sh`, `lib/config.sh`, `lib/reaper.sh`, `lib/adapters/hitl/_registry.sh`. Re-sourcing is idempotent for these files.

3. **Fixed the record-replay harness** — same bug class existed in `tests/integration/lib/record-replay.sh`. Plus 4 other bugs:
   - `RECORD_REPLAY_CURRENT_FILE` wasn't exported by `sparc_rr_init`
   - Bash-interpolation of `$RECORD_REPLAY_CURRENT_INDEX` in python produced `idx =` (syntax error)
   - `RECORD_REPLAY_CURRENT_FILE` reset to empty in record-replay.sh defaults, clobbering parent value
   - Playback index was in-process only; didn't survive the runner's exec'd children. Fixed by persisting to a `.idx` file.

4. **Recorded real Hermes sessions** — `tests/integration/fixtures/setup_against_hermes.json` now has 6 real recorded interactions against real Hermes v0.17.0:
   1. `kanban boards list`
   2. `kanban boards create sparqr-itest-replay --name ... --icon 🎯 --switch`
   3. `kanban --board sparqr-itest-replay create [SPEC] test goal`
   4. `kanban --board sparqr-itest-replay create [UREFINEMENT] ... --parent ...`
   5. `kanban --board sparqr-itest-replay link ... ...`
   6. `kanban --board sparqr-itest-replay comment ... ...`

5. **Verified end-to-end** — both RECORD mode and REPLAY mode of the integration test pass 4/4.

## v0.2.1's value vs v0.2.0's

This is what v0.2.1 ACTUALLY delivers, beyond what was planned:

1. **Real-Hermes compatibility verified.** v0.2.0 said "tested against Hermes v0.51.0" but it was actually built against a hypothetical verb set. v0.2.1 is tested against real Hermes v0.17.0 and the verbs match.

3. **A real bug fixed.** The double-source guard pattern affected 7 lib files plus the record-replay harness. Without the guard fix, **v0.2.0 was broken in production** — every `sparc <subcommand>` invocation failed because the dispatcher `exec`'d subcommands and lost function definitions. This bug was masked in unit tests because they source libs directly without going through the dispatcher.

This last point is the biggest finding. v0.2.0's "237 passing unit tests" was a false positive — the tests didn't exercise the bug because they bypassed the dispatcher. v0.2.1 caught and fixed this. **The user was right to insist on testing against an actual hermes instance** — that's what surfaced this bug.
