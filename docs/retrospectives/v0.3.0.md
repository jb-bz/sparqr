# Retrospective: v0.3.0

## What we said we'd do

v0.3.0 — "Make it pleasant" — was scoped at 28 points across 8 stories (after splitting the 13-pt gates story into 5 sub-stories):

1. `sparc status` command (3 pts)
2. Structured HITL gate types (13 pts — split into 1a-d + 1e)
3. JSON schema for `sparc.config.yaml` (3 pts)
4. Artifact reconciler (5 pts)
5. Log rotation (2 pts)

Plus smaller sub-stories that fell out during planning: OrbStack/runtime selection was already done in v0.2.1.

All 8 stories shipped. **Estimated and actual: 28 pts. Velocity ratio: 1.00.**

- ✅ **`sparc status`** — boards / counts / running / blocked view; `--board` filter; `--json` output
- ✅ **Gate types** — `approval | confidence | sampling | exception` with config-driven thresholds

- ✅ **`sparc config show`** — pretty-prints parsed config
- ✅ **Artifact reconciler** — background sync of disk artifacts → kanban comment threads; idempotent
- ✅ **Log rotation** — `sparc-pipeline.log` rotates at 50MB, keeps last 5 gzipped

## What surprised us

1. **Tests-first caught real bugs.** Story 1a's gate parser test failed until I added `source "$(dirname "${BASH_SOURCE[0]}")/config.sh"` to `lib/gates.sh` — the test couldn't have caught that without writing the test first. This is the second cycle in a row where tests-first paid off (v0.2.1's record-replay harness had similar findings).
2. **bash 3.2 quirks are real and recurring.** v0.3.0 hit 4 different bash 3.2 / macOS awk issues:
   - `mapfile` not available
   - gawk `match()` with capture groups not supported on macOS awk
   - `for arg in "$@"` + `shift` interact badly with each other
   - bash function source ordering: a stub override BEFORE sourcing is clobbered by re-sourcing
   These are exactly the same class of bugs we hit in v0.2.1. Worth investing in a `lib/bash3-compat.sh` shim for v0.4.0.
3. **Hermes events store metadata, not content.** When trying to grep comment threads for the `[RECONCILED:hash]` marker, I discovered that Hermes stores `{author, len}` only — not the actual comment text. This is a real Hermes limitation that affects how we should design durable state. Local state files are the right answer for v0.3.0.
4. **The "deprecate `requires_review` boolean" story was anticlimactic.** I expected this to be a big refactor. Instead: keep the boolean (it defines the default), let the gates config override it. The orchestrator's `if [[ "$gate_decision" == "auto-approve" ]]` short-circuit means v0.2.0 behavior is fully preserved when no `gates:` config is present. Zero migration friction for existing users.
5. **Integration tests are still mostly placeholders.** v0.3.0 added 0 new integration tests (we have 3 from v0.2.1). The integration test framework works, but I haven't added tests that exercise the new gates/reconciler/logrotate logic. That's a v0.3.1 candidate.

## What we'd do differently

1. **Add integration tests for new commands.** `sparc status`, `sparc config validate`, `sparc reconciler`, `sparc logrotate` all have unit tests but no real-Hermes integration tests. The pattern from v0.2.1 (record-replay against real Hermes) should be applied to all new commands. v0.3.1.
2. **Build a `lib/bash3-compat.sh` shim.** We've now hit bash 3.2 / macOS awk issues 8+ times across v0.2.1 and v0.3.0. Time to centralize the workarounds.
3. **Story 8's bug discovery was lucky.** The JSON schema caught a real bug in `sparc.config.yaml.example`, but only because I ran `sparc config validate` against it during story development. Future versions should run `sparc config validate` as part of CI / pre-commit.
4. **Document the bash source-order trap in a skill.** This came up in story 6 testing — future tests that mock `sparc_kanban_event_log` need to know to override AFTER sourcing. Could be a `sparc-testing-patterns` skill.
5. **The reconciler's verification was manual.** I verified it end-to-end against real Hermes (manually), but there's no automated test that catches if Hermes changes its comment-storage behavior. Worth a story in v0.3.1: "add integration tests for reconciler that exercise real Hermes comment posting."

## What this means for v0.4.0

- The bash 3.2 / macOS awk bugs are a recurring cost. v0.4.0 should include a small (1-2 pt) story to consolidate them into `lib/bash3-compat.sh`.
- The integration test framework exists but isn't being used for new commands. v0.4.0's first story should be "add integration tests for status, config, reconciler, logrotate" so future versions catch breakage end-to-end.
- The gates + reconciler + logrotate + status combination creates a workflow that's actually pleasant to use. v0.4.0 should focus on adoption (new-project template, hosted demo, video walkthrough) so other people can discover this.
- Velocity is 1.00 for v0.3.0 (matched estimate exactly). v0.4.0 is estimated at 36 pts; if v0.4.0 ships at 36+, our point scale is well-calibrated. If it ships at 50+, we're under-estimating again.
