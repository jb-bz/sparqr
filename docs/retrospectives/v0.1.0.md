# Retrospective: v0.1.0

**Released:** 2026-06-17 (announced) / 2026-06-18 (4 commits to main)
**Status:** Initial public release.

> ⚠️ **This release predates the story-point scale.** The v0.1.0 retrospective is partially reconstructed after the fact. Future retrospectives (v0.2.0+) will have actual story-point tracking from the start, making them more accurate. We're starting from v0.2.0.

## What we said we'd do

The v0.1.0 spec was negotiated in conversation before the point scale existed. The agreed scope:

- 6-stage SPARC+Design pipeline (Spec → Design → Pseudo → Arch → Refine → Complete) with the Design phase as a community extension
- 7 named Hermes profiles (6 stage agents + 1 reviewer)
- Hermes Kanban as the durable coordination substrate
- 5 pluggable HITL adapters (terminal, tui, webui, workspace, official-dashboard)
- 5 skills for the running Hermes
- 6 artifact templates
- A `sparc` CLI with the full subcommand surface
- `sparc-pipeline` orchestrator daemon
- Full docs (INSTALL, ARCHITECTURE, HITL, ADDING-STAGES, TROUBLESHOOTING, FAQ)
- 111 tests across 5 suites, all passing
- Public release on GitHub

## What we actually shipped

All of the above, plus several things we hadn't planned:

- ✅ Bitwarden Secrets Manager integration (Hermes first-party plugin) — discovered mid-project when designing the secret-handling architecture
- ✅ Per-stage model routing (originally planned for v0.2.0) — moved up because the cost savings are obvious
- ✅ BSM-aware push to GitHub (the user's actual deployment workflow, which the package had to support)
- ✅ `sparq`/`sparqr` naming — chose the brand during the conversation
- ✅ Documentation polish (badges, hero section, ASCII architecture diagrams, comparison table) — added in a second pass after initial release
- ✅ `ROADMAP.md` with critical review and versioned plan — added after the release as a follow-up
- ⚠️ Notify channel (Telegram/Discord/Slack) — explicitly dropped from v0.1.0 per user feedback ("I want user preference, not a Telegram channel hardcoded")

**Total retroactive estimate: ~55 pts**

**What surprises me:** the orchestrator (`sparc-pipeline`) and the docs are roughly the same size (5 pts each). The orchestrator is the most "code-heavy" thing in the package, and the docs are roughly the same effort. This validates a suspicion I had: for a tool like this, the docs are half the work.

## What surprised us

1. **BSM was harder than expected.** The `bws` CLI's "Doesn't contain a decryption key" error took meaningful debugging time. The work-around (the hermes cache) works but is non-obvious. We documented it in TROUBLESHOOTING but the underlying issue (a v1/v2 token format mismatch in Bitwarden's tooling) is upstream of us.

2. **Function-hoisting bug in bash.** I wrote `sparc-pipeline` with function definitions below the case statement that called them. Bash doesn't hoist function definitions like JavaScript. Cost: one bad user-facing test failure, ~1 pt of debugging. **Lesson: bash scripts that dispatch on `$1` need function definitions BEFORE the case statement.** This is now a documented pattern in the package.

3. **The "Design phase" debate was the right call.** Upstream SPARC has 5 phases. Adding Design as a 6th was a user decision. The package's value proposition is partially the 6th stage. Future releases should preserve it as a default but make it easy to disable (per-project config).

4. **The notify channel was the right thing to drop.** I tried to build a Telegram notifier, then realized the user wanted it to be pluggable and user-preference-driven, not hardcoded. Dropping it from v0.1.0 was the right call — adds noise to ship it for a user who said "I have many chat gateways, I want choice."

5. **The test suite caught a real bug.** The function-hoisting bug (above) was caught by `test_e2e.sh` before the package ever shipped. Without the integration test, it would have been a "weird bug" report in the first user issue. Worth its weight.

## What we'd do differently

1. **Story points from day one.** Even if the scale was rough at first, having actuals to compare to estimates is the only way to calibrate. v0.2.0 will have proper points.
2. **A `sparc doctor` step in `setup.sh`** before the heavy install. We added this to the v0.2.0 roadmap (story 8) because the install experience is too dependent on the user already having Bitwarden + Hermes set up.
3. **Type the `sparc-pipeline` flow more carefully.** The bash function-hoisting bug is embarrassing; it's exactly the kind of thing a typed language would have caught. v0.2.0's event-based poller should be tested more rigorously.
4. **Decide on the v0.2.0 scope before v0.1.0 ships.** We added 9 stories to v0.2.0 in the post-release roadmap. Some of those should have been v0.1.0. Specifically: the per-stage model routing is high-value, low-cost, and we deferred it. Retroactively, it should have been v0.1.0.

## Velocity data (for future calibration)

- **Calendar time of git activity:** 1 day (all 4 commits on 2026-06-18)
- **Total conversation span:** several days (planning, building, debugging BSM, docs polish, roadmap, story points)
- **Story points shipped:** ~55 (retroactive)
- **Implied velocity:** too noisy to be useful (calendar time is misleading; conversation span is more honest but not measurable in a clean way)

For v0.2.0 onwards, we'll measure velocity as: "story points per *commit cluster* that says `Release v0.X.0`." That gives us a clean unit: a release is a set of commits; a release has a point total; we can compare release-to-release ratios.

## What this means for v0.2.0

- v0.2.0 is estimated at 46 pts across 9 stories. If our v0.1.0 velocity is approximately 55 pts per release (one git-day), v0.2.0 is a single release.
- However: v0.1.0 didn't include real Hermes integration testing. v0.2.0 will. The integration test suite alone is 8 pts. If integration tests surface unexpected issues (they will), v0.2.0 may stretch to 60-70 pts.
- Realistic v0.2.0 expectation: 50-60 pts actual. Anything in that range is healthy. If we hit 80+, we should split v0.2.0 → v0.2.0 + v0.2.5.
- The user will be prompted to write the actual retrospective when v0.2.0 ships; we'll use this template as a starting point.
