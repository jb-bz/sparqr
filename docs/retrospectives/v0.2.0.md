## What surprised us

Five specific things:

1. **BSM was harder than expected.** The `bws` CLI's "Doesn't contain
   a decryption key" error took meaningful debugging time during the
   initial package work. The work-around (the hermes cache at
   `~/.hermes/cache/bws_cache.json`) works but is non-obvious. The
   underlying issue (a v1/v2 token format mismatch in Bitwarden's
   tooling) is upstream of us. Documented in TROUBLESHOOTING.md but
   the underlying fix is on Bitwarden.

2. **The function-hoisting bug in bash was embarrassing.** During
   initial work, the orchestrator (`sparc-pipeline`) had function
   definitions below the case statement that called them. Bash
   doesn't hoist function definitions like JavaScript. Caught by
   `test_e2e.sh` before shipping — that's the value of the e2e test.
   Lesson: bash scripts that dispatch on `$1` need function
   definitions BEFORE the case statement.

3. **The notify channel drop was the right call.** The original
   package had a Telegram notifier (v0.1.0). The user pushed back
   ("I have many chat gateways, I want choice, not a hardcoded
   Telegram"). Dropping it was the right call — adds noise to ship
   for a user who said "user preference, not Telegram default."

4. **The design phase is the package's value proposition.** Upstream
   SPARC has 5 phases; we added Design as a 6th. The package's
   differentiation is partially this 6th stage. Future releases
   should preserve it as a default but make it easy to disable
   (per-project config).

5. **Story 5's preflight test bug was found and fixed mid-work.**
   The `tests/test_preflight.sh` was checking the function's
   behavior against the dev env's actual state, which assumed
   hermes wasn't on PATH. Once hermes WAS on PATH, the test
   started failing. Fixed by stripping PATH inside the test only.
   Cost: 5 min. Lesson: tests that depend on environment state are
   flaky by definition; strip the environment, don't rely on it.

## What we'd do differently

Five specific changes for v0.3.0:

1. **Define the YAML schema explicitly before implementing.** Story 5
   took 3 commits because the parser's design evolved (top-level
   scalars vs sub-maps were a late realization). For v0.3.0, write
   the schema docs first, then implement against them.

2. **Story 6 should have been planned as a 2-phase story.** The
   "build the framework" and "record real sessions" are different
   kinds of work; bundling them into one story made it hard to
   know when v0.2.0 was "done." For v0.3.0: phase 1 = framework
   (committable), phase 2 = first recorded session (separate
   commit, separate story).

3. **Tests-first debugging paid off again.** Story 4's tests
   caught a missing-frontmatter bug in the new skill before any
   commit was made. Story 5's tests caught the parser design bug
   (top-level scalars). The pattern is: write the test, watch it
   fail, fix the bug. Repeat.

4. **Make "scaffolded" a first-class status.** Story 6 ships with
   the framework but no real recordings. The ROADMAP marks it as
   "Done (scaffolded)" with an honest caveat. That language is
   useful; the v0.1.0 retrospective should adopt it.

5. **Document env-leak hazards.** Three bugs in v0.2.0 came from
   env vars leaking between test runs (SPARC_KANBAN_LOADED,
   SPARC_VALIDATORS_LOADED, SPARC_STAGES_LOADED, SPARC_CONFIG_LOADED).
   Each test now unsets these at the top, but the pattern is
   fragile. v0.3.0 should add an `lib/test-isolation.sh` helper
   that handles this centrally.

## What this means for v0.3.0

- v0.3.0 is scoped as "Make it pleasant" (28 pts in the current
  ROADMAP). That's roughly 70% of v0.2.0's 40 pts. Realistic.
- Story 6's deferred work (recording real sessions) should be a
  v0.2.1 task, not v0.3.0. Don't mix it with new features.
- Story 1's "PID file + time fallback" gap (per ROADMAP story 3.0
  notes) should be a v0.3.0 story. The current PID-only approach
  doesn't catch alive-but-stuck agents.
- The user's "story points only, no human time" rule was honored
  throughout v0.2.0. Continue this for v0.3.0.
- The retrospective system itself is working: this is the second
  one written (after v0.1.0's), and the format has held up.
